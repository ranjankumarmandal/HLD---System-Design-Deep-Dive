Functional Requirement:
-----------------------
- Receives new order requests, including items, costomer location, delivery preferences
- Returns possible shipment options for an order (cost, estimated delivery, split shipments, etc.)
- Allocates the item to warehouses and cretes shipment tasks
- Updates warehouse inventory as items are allocated
- Provides shipment status to customers or internal tools
- For updating warehouse / carrier metadata, prices, or capacity

Non-Functional Requirement:
---------------------------
- Low latency
- High availability
- Massive Scale
- (And so on...)

Assumptions:
-------------
- Traffic estimates: Amazon processes millions of orders per second, so the system must handle thousands of requests per second (peak traffic cold be 10K+ rps during events like prime data)
- Spike handling: Using automatic scaling for stateless services (Kubernetes / EC2 auto-scaling), and queueing for bust traffic (order requests can be queued for allocation)
- Parallel processing: Allocation logic can be parallelized by region or warehouse

API Design:
------------
POST /allocation/createOrder
GET /allocation/getShipmentOptions
POST /allocation/allocateShipment
PUT /allocation/updateInventory
GET /allocation/trackShipment
GET or POST or PUT /allocation/admin
