Functional Requirement:
-----------------------
- Inventory Management - Add, update, remove, query inventory
- Receiving shipments
- Picking & Packing
- Shipping
- Return processing
- Storage optimization
- User authentication
- Dashboard analytics
- Error / incident logging

Non-Functional Requirement:
---------------------------
- Low latency
- High Availability
- Massive scale
- (and so on...)

Assumptions:
-------------
- Inventory Operations - If this warehouse handles 1M items, with 100K actions / day, thats about 1 to 2 request / sec average, but can spike to 100+ request / sec during peak
- Picking / Packing - Hundreds of worker / robots may work in parallel, so parallelizing is key here. Assuming 500 concurrent users / robot
- Analytics / Dashboard - Less frequent, but potentially heavy queries
- Peak load - upto 1K request / sec per warehouse, scaling horizontally. Off peak - 50 - 100 request / sec

DB Schema:
----------
- Inventory DB: RDBMS: PostgreSQL: (Inventory: item_id(PK), name, quantity, location_id, status, last_updated), (Location - location_id(PK), warehouse_id(FK), zone, shelf, bin)
- Operations / Even Log DB: NoSQL: DynamoDB: (OperationsLog: operation_id(PK), timestamp, item_id(FK), type, user_id, status, notes)
- Warehouse Layout DB: RDBMS: Graph DB: (....)
- User DB / Robot DB: RDBMS: (Users: user_id(PK), name, role, warehouse_id(FK))

HLD:
----
Client (Optional) -> API Gateway -> Dashboard Service + Picking / Packing Service + Shipping Service + Receiving Service + Inventory Service -> OperationLog DB + Inventory DB
