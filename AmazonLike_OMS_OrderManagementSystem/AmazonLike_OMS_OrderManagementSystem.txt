Functional Requirement:
---------------------------
- Place a new order
- Retrieve details about an order
- Update existing order
- Cancel an order
- List all orders to an user.
- Initiate a return (optional)
- Check the status of order


Non-Functional Requirement:
-----------------------
- Low latency
- High availability
- Massive scale
- (and so on...)



Assumptions:
-------------
- 10k orders / sec: on regular days
- 100k orders / sec: on peak or prime days
- 10x reads per write (browsing order history, tracking)

- EKS
- Message queues (for heavy writes)
- Caching for hot data (tracking, order summaries)


API Design:
------------
POST /orders
GET /orders/{orderId}
PUT /orders/{orderId}
DELETE /orders/{orderId}
GET /orders?userId={userId}
POST /orders/{orderId}/return
GET /orders/{orderId}/tracking


DB Schema:
-----------
- Order Metadata: RDBMS
-- Orders: RDBMS: (order_id, user_id, status, created_at, shippping_address, payment_status, items, total_price)
-- Order Items: (order_id, item_id, quantity, unit_price)
-- Order Events: (event_id, order_id, event_type, timestamp, metadata)
- Cache: Radis (for hot orders)
