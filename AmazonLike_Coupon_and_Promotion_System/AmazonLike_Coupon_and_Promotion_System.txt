Functional Requirement:
-----------------------
- Create a new coupon
- Update an existing coupon
- Delete a coupon
- List available coupons for a user
- Redeem a coupon during checkout
- Get user's redeemed/expired coupons
- Create a new promotion
- List current promotions
- (Optional): Coupon & promotion analytics

Non-Functional Requirement:
---------------------------
- Low Latency
- High Availability
- Massive Scale
- (and so on...)

Assumptions:
------------
- Assuming: 10M active users, ~1M orders / day (~11. orders / sec)
- During Spikes: 10x: 100 orders / sec: same for coupon validation
- Admin Actions: 1 / sec. Coupons Browsing: 1000 / sec during sales

API Design:
-----------
POST /admin/coupons
PUT /admin/coupons/{couponId}
DELETE /admin/coupons/{couponId}
GET /coupons
POST /coupons/redeem
GET /coupons/history
POST /admin/promotions
GET /promotions
GET /admin/reporting

DB Schema:
----------
- RDBMS: PostgreSQL:
---- Coupon (coupon_id(PK), code, description, discount_type, discount_value, expiry_date, usage_limit, created_at, updated_at)
---- Promotion (promotion_id(PK), title, description, start_date, end_date, discount_type, discount_value, applicable_product_ids, applicable_categories_ids, created_at, updated_at)
---- UserCoupon (user_coupon_id(PK), user_id, coupon_id, redeemed_at, status)
---- CouponRedemption (redemption_id(PK), user_id, coupon_id, order_id, redeemed_at)

- NoSQL DB: Radis: for fast lookup and caching of available coupons / promotions
