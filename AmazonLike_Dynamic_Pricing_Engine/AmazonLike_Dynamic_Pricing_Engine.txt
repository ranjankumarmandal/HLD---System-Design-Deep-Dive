Functional Requirement:
-----------------------
- Get the current price for a product
- Update the price for a productt
- Create, update or delete rules / algorithms for dynamic pricing
- Fetch historical prices for a product
- Accept competitor price feeds
- Update inventory levels for products
- Query audit logs for price changes

Non-Functional Requirement:
---------------------------
- Low latency
- High availability
- Massive scale
- (And so on...)

Assumptions:
------------
- Assume: 10M products, each may require price updates every hour (peak: flash sales)
- At peak: 10M price update request / hour. 2.8K updates / sec
- Price queries could be much higher, especially during big events (Black Friday): Assume 100K queries / sec
- System needs to parallelised price update calculation and ingestion, w will use batch processing during low load, real-time streaming during high load

API Design:
------------
GET /price/{productId}
PUT /price/{productId}
GET, POST, DELETE /price/dynamic
GET /price/history/{productId}
POST /price/competitor
PUT /inventory/{productId}
GET /price/analytics

DB Schema:
----------
- Primary DB: Distributed NoSQL DB: Cassandra: For product prices: (Product_Price: product_id (PK),current_price, last_updated, price_source)
- Rules DB: RDBMS: PostgreSQL: For pricing rules: (Pricing_Rules: rule_id, product_id, rule_type, parameters, active)
- Audit / History DB: Time-series DB: TimescaleDB: For price history and audit logs: (Price_History: product_id, price, timestamp, reason), (Competitor_Price: product_id, competitor_id, competitor_price, fetched_at)

HLD:
-----
Client -> API Gateway -> Pricing Service + Monitoring Logging -> Rule Engine + Inventory Service + Compliance Data Service + Price History Service + Product Prices DB + Cache + Notification Service + Batch Processing Workers -> Price Rule DB + Competitor Prices DB + Price History DB
