Functional Requirement:
-----------------------
- Returns a list of recommended product IDs for a given user
- Receives activity events like clicks, purchases, views, and updates user profiles
- Returns a list of similar products for a given product
- Add / update a product in the catalog
- (Optional): Returns trending products

Non-functional Requirement:
----------------------------
- Low latency
- High availability
- Massive scale
- (and so on...)

Assumptions:
------------
- Assume: 100M users, 10M products
- Peak load: 10k req / sec for recommendations during events / holidays
- User activity events: upto 100k / sec at peak
- Storage: Billions of activity events, millions of profile updates daily
- To handle spikes, we are going to use autoscaling and batch processing for heavy computations
- Caching layers to absorb read spikes for popular products / users

API Design:
-----------
GET /recommendations?userId=xxx
POST /userActivity
GET /products/{productId}/similar
POST /product
GET /recommendations/trending

DB Schema:
----------
- User Activity Store: NoSQL: Cassandra: (UserActivity: userId, eventType, productId, timestamp, metadata)
- User Profile Store: NoSQL: DynamoDB: (UserProfile: userId, recentViews, purchaseHistory, preferences: lastUpdate)
- Product Catalog: RDBMS or NoSQL postgreSQL, DynamoDB : (Product: productId, title, description, category, price, tags, imageUrl, availability)
- Recommendation Cache: Radis (key: userId, value: recommendedProductIds)
- Model: S3 / GCS: Blob storage for ML models / artifacts
