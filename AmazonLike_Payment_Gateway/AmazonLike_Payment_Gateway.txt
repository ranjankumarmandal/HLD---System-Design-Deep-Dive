Functional Requirement:
-----------------------
- Initiate a payment transaction
- Confirm payment after OTP authentication
- Check the status of a transaction
- Request refund for a transaction
- Endpoint for external payment provider callbacks
- Retrieve a users payment history
- (Optional): Analytics for admins / merchants

Non-functional Requirement:
---------------------------
- Low Latency
- High Availability
- Massive Scale
- (and so on...)

Assumptions:
------------
Traffic Estimation:
- Assume 10M active users, with peak 10K transactions / sec during sales
- Average Load: 2K transactions / sec
- Each transaction may require 3-5 api calls
- Need to handle traffic spikes - auto-scaling and queueing for burst traffic

Handling Peaks:
- Use load balancers, auto-scaling groups
- Queue incoming requests if backend is overloaded
- Design for eventual consistency for non-critical updates

API Design:
-----------
POST /payments/initiate
POST /payments/confirm
GET /payments/status/{id}
POST /payments/refund
POST /payments/webhook
GET /payments/history/{userId}
GET /payments/analytics

DB Schema:
----------
- Transactional DB: PostgreSQL: Store payments, users, transaction logs: ({payments: id(PK), user_id, amount, currency, status, payment_method, created_at, updated_at, provider_txn_id}, {payment_methods: id(PK), user_id, type, token, expiry, is_default, created_at}, {refunds: id(PK), payment_id (FK), amount, status, created_at, processed_at}, {users: id(PK), names, email, phone, created_at})
- NoSQL DB: MongoDB or DynamoDB: For high velocity session data or caching
- Cache: Radis: For fast access to transaction / session states
- Data Warehouse: BigQuery: Analytics and reporting

HLD:
----
Client -> API Gateway -> Payment Service + Payment Method Service -> Transaction Service + Refund Service + Notification Service + Fraud Detection Service + External Provider -> DB + Cache + Queue
