Functional Requirements:
------------------------
- Main endpoint to accept search queries with optional filters... etc.
- Return search suggestions as user types
- Accepts catalog changes
- For monitoring service health
- ....


Non-Functional Requirements:
----------------------------
- Low latency
- High Availability
- Massive scale
- (and so on...)


Assumptions:
------------
- Traffic load: peak 10M QPS (query per second) during major sales, normally around 2M QPS.
- Index updates: 100k products / sec during peak inventory changes
- Reads are much higher than writes
- Concurrency: Scaling horizontally. Each search nodes can handle 10k QPS, so during peak hours we are going to need thousands of nodes
- Handling spikes: Deploy autoscaling and rate limiting for protection against DDoS and traffic surges


API Design:
-----------
POST /search
POST /autocomplete
POST /index
GET /health


DB Schema:
-----------
- NoSQL: Document-oriented search DB: ElasticSearch, OpenSearch, Slr, ...etc
-- Search DB: (product_id, title, description, price, category, brand, attributes, images, inventory, rating, updated_at)
- RDBMS/NoSQL: Operational Metadata: (user_id, search_history.preferences)
- Cache (Radis, ... etc)
