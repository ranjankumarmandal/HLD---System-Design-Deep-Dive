Functional Requirement:
-----------------------
- Start the checkout process, lock cart, prepare summary
- Apply a coupon or promotion code
- Select shipping option
- Submit payment details
- Finalize and place the order
- Get the current checkout summary for review
- (Optional): Retrieve past checkouts

Non-functional Requirement:
---------------------------
- Low latency
- High availability
- Massive scale
- (and so on...)

Assumptions:
------------
- Suppose we expect 10k checkouts / sec during peak sale events
- Each checkout might generate 4 - 5 API requests. So about 40K - 50K RPS
- Inventory and payment are bottlenecks and need to be heavily parallelized
- During off-peak hours: load may drop to 1K RPS
- We need auto-scalling and rate limiting to manage surges

API Design:
-----------
POST /checkout/start
POST /checkout/apply-coupon
POST /checkout/select-shipping
POST /checkout/payment
POST /checkout/confirm
GET /checkout/summary
GET /checkout/history

DB Schema:
----------
- Cart/Checkout State: Radis (key: cart_id, value: user_id, items, expires_at)
- Order Data: RDBMS: PostgreSQL (Order: order_id (PK), user_id (FK), items, total_price, shipping_address, payment_status, created_at, status)
- Inventory: RDBMS: PostgreSQL: (Inventory: product_id (PK), quantity_available)
- Product / Price: NoSQL: MongoDB
- Audit / History: Append-only log: S3, Kafka, BigQuery

HLD:
----
Client -> API Gateway -> Checkout Service -> Pricing Service + Cart Service + Order Service + Order DB + Payment Service + Inventory Service -> Inventory DB + Cart DB + Payment Provider + Notification Service
