Functional Requirement:
-----------------------
- Warehouse clients send stock changes
- Query the current global stock for one or more items
- Query stock levels for a specific item at a specific warehouse
- Get the history of changes for an item

Non-functional Requirement:
---------------------------
- Low latency
- High availability
- Massive scale
- (And so on...)

Assumptions:
------------
- Assume: 1000 warehouses, each handling 100 stock changes / sec (sales, inbound, returns, etc.)
  -- Total write traffic: 1000 * 100 = 100K writes / sec
- Read queries: (for order placement, internal tools): Could spike to 1M / sec globally, especially during sales
- During peak (Black friday, prime day): Both reads and writes may spike 10x
- To handle bursts, we'll need horizontal scalability (sharding, load balancing, and message batching)

API-Design:
-----------
POST /stock/update
GET /stock/query
GET /stock/warehouse
GET /stock/history
