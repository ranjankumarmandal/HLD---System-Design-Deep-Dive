Functional Requirement:
-----------------------
- Add a new product
- Update product details
- Retrieve product by id
- Search / filter products by attributes
- Remove product
- (Optional): Product history

Non-Functional Requirement:
---------------------------
- Low latency
- High availability
- Massive scale
- (And so on...)

Assumptions:
------------
- Reads: Millions of request / sec
- Writes: Thousands / sec
- Assume: 100M products, peak 10M reads / sec, 100K writes / sec
- During slow time: Reads may drop by 1M / sec, writes drop to 10K / sec
- To handle spikes, auto-scaling groups and sharded db wil be used

API Design:
-----------
POST /products
PUT /products/{id}
GET /products/{id}
GET /products/search
DELETE /products/{id}
GET /products/history

DB Schema:
----------
- Primary DB: Distributed NoSQL: Cassandra or MongoDB: For horizontal scaling, fast reads / writes, flexible schema: (Product: product_id (PK), name, description, brand, category, price, attributes, stock, seller_id, created_at, updated_at)
- Secondary DB: Elasticsearch DB: For fast full-text and attribute search / filtering: (Elasticsearch Index: product_id, name, branch, category, price, attributes)
- Cache: Radis: For frequently accessed products and search results: (key: product_id, value: product data or search results)

HLD:
----
client -> API Gateway + Seller Service + Monitoring Service -> Product Catalog Service + Replication / Backup Service -> Search Engine + NoSQL DB + Cache
